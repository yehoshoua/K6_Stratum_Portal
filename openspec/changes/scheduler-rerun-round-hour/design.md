## Context

The backend already has an internal scheduler that polls `test_schedules` and launches K6 TestRun CRDs via `runScheduleTestRun`. Run names are generated by `buildTestRunNames`, which currently uses an underscore-formatted timestamp. Schedules can be created with any valid 5-field cron expression, and the UI does not enforce a round-hour start. The change focuses on guaranteeing a consistent naming format for scheduler-triggered reruns and enforcing round-hour cron expressions.

## Goals / Non-Goals

**Goals:**
- Ensure scheduler-triggered reruns use the original test name plus a `yyyy-mm-dd-hh-mm` suffix.
- Enforce round-hour cron expressions (minute field `0`) when creating or editing schedules.
- Keep the scheduling engine and storage model intact.

**Non-Goals:**
- Introducing new scheduling backends or external job queues.
- Migrating existing schedule data or rewriting existing run history.
- Changing how manual, non-scheduled runs are created beyond naming alignment decisions.

## Decisions

1. **Reuse existing scheduler flow; adjust naming format centrally.**  
   Update the timestamp format used by `buildTestRunNames` and its suffix regex so scheduler reruns (and any run paths that share this function) consistently append `yyyy-mm-dd-hh-mm`.  
   *Alternative:* Only customize scheduler-run naming. Rejected because it would split naming logic and invite divergence between scheduled and manual runs.

2. **Round-hour validation in UI + API.**  
   Enforce the minute field equals `0` in the UI for immediate feedback, and in the schedule create/update API to ensure consistent behavior for non-UI clients.  
   *Alternative:* UI-only validation. Rejected to prevent inconsistent data when schedules are created via API or future automation.

3. **Preserve the original schedule name as the base for rerun names.**  
   The schedule name becomes the “original name” anchor, and the suffix is appended during execution.  
   *Alternative:* Store a separate original name field. Rejected to avoid schema changes.

## Risks / Trade-offs

- **Run-name format change affects downstream tooling** → Communicate the new suffix format; ensure `stripTestRunDateSuffix` is updated to match so UI grouping stays consistent.
- **Round-hour enforcement could reject existing cron patterns on edit** → Validation only runs on create/update; existing schedules remain unchanged until edited.
